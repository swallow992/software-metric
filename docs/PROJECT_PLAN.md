# 软件度量自动化工具项目文档

## 1. 项目目标

本项目计划开发一个中小型软件度量自动化工具，将软件度量课程中的基本原理、模型和方法应用到真实软件项目分析中。

工具应支持对 Java 源代码和用户输入数据进行自动化度量，输出代码规模、复杂度、面向对象度量、工作量、成本、开发时间和人员需求等结果，并生成适合课程设计、实验报告和项目演示使用的分析报告。

## 2. 项目定位

项目类型：课程设计型软件工程工具。

核心用户：

- 软件度量课程学习者
- 软件质量保证课程学习者
- 需要分析 Java 项目规模、复杂度和面向对象质量的开发者

核心价值：

- 自动解析 Java 项目源码
- 计算常见软件度量指标
- 支持面向对象度量模型
- 支持成本、工期、人员等估算
- 提供可视化结果和规范报告

## 3. 技术路线

优先采用以下技术方案：

- 后端：Java 17、Spring Boot、Maven
- 源码解析：Eclipse JDT Core ASTParser
- 前端：Thymeleaf、HTML、CSS、JavaScript
- 图表展示：ECharts
- 报告导出：Markdown、HTML、JSON
- 单元测试：JUnit

选择 Java 和 Eclipse JDT 的原因：

- README 中明确建议使用 Eclipse ASTParser 获取 Java 程序 AST
- Java 项目适合展示面向对象度量
- Spring Boot 便于快速开发可演示的 Web 工具
- Maven 便于管理 JDT、测试和报告相关依赖

## 5. 软件度量指标设计

### 5.1 代码规模度量

需要实现的指标：

- 总行数
- 空行数
- 注释行数
- 有效代码行数
- 文件数量
- 包数量
- 类数量
- 接口数量
- 方法数量
- 字段数量

### 5.2 复杂性度量

需要实现的指标：

- 方法圈复杂度
- 类平均圈复杂度
- 类总圈复杂度
- 高复杂度方法数量
- 高复杂度类数量

圈复杂度基础规则：

- 方法默认复杂度为 1
- 遇到 `if`、`for`、`while`、`do while`、`case`、`catch`、条件表达式、逻辑与、逻辑或时增加复杂度

### 5.3 CK 面向对象度量模型

优先实现以下 CK 指标：

- WMC：Weighted Methods per Class，类中方法复杂度总和
- DIT：Depth of Inheritance Tree，继承树深度
- NOC：Number of Children，直接子类数量
- CBO：Coupling Between Object Classes，类之间耦合数量
- RFC：Response For a Class，类可响应的方法集合数量
- LCOM：Lack of Cohesion in Methods，方法内聚缺失度

### 5.4 LK 或扩展面向对象度量

在 CK 指标基础上扩展实现：

- 类规模
- 属性数量
- 方法数量
- 公有方法数量
- 私有方法数量
- 继承关系数量
- 依赖关系数量
- 聚合或组合关系数量
- 类职责复杂度

如果后续发现课程对 LK 模型有明确公式，应优先按课程公式修正文档和实现。

### 5.5 估算类度量

计划支持以下估算：

- 功能点 FP
- 用例点 UCP
- COCOMO 工作量估算
- 成本估算
- 开发时间估算
- 人员数量估算

估算模块需要支持用户输入参数，例如：

- 外部输入数量
- 外部输出数量
- 外部查询数量
- 内部逻辑文件数量
- 外部接口文件数量
- 用例数量和复杂度
- 参与者数量和复杂度
- 人员月成本
- 生产率参数

## 6. 系统模块规划

建议项目包结构：

```text
software-metric
├── src/main/java
│   └── com/example/softwaremetric
│       ├── SoftwareMetricApplication.java
│       ├── controller
│       ├── service
│       ├── parser
│       ├── metric
│       ├── model
│       ├── report
│       └── util
├── src/main/resources
│   ├── templates
│   ├── static
│   └── application.yml
├── src/test/java
├── examples
├── docs
└── README.md
```

模块职责：

- `controller`：处理页面请求、上传请求和报告导出请求
- `service`：组织完整分析流程
- `parser`：负责 Java 源码读取和 AST 解析
- `metric`：负责各类指标计算
- `model`：保存分析结果、类信息、方法信息和估算参数
- `report`：生成 Markdown、HTML、JSON 报告
- `util`：通用工具方法

## 9. 数据流设计

基础分析流程：

```text
用户选择源码
    ↓
扫描 Java 文件
    ↓
读取源码内容
    ↓
ASTParser 解析
    ↓
提取类、方法、字段、依赖和继承信息
    ↓
计算基础指标、复杂度指标和面向对象指标
    ↓
生成分析结果对象
    ↓
页面展示或报告导出
```

估算流程：

```text
用户填写估算参数
    ↓
参数校验
    ↓
功能点、用例点或 COCOMO 计算
    ↓
输出工作量、成本、工期、人员估算
```

## 10. 风险与约束

技术风险：

- Eclipse JDT ASTParser 配置较繁琐，需要处理源码路径、编码和 Java 版本
- CK 指标中的 CBO、RFC、LCOM 需要结合 AST 信息合理近似实现
- 类图、流程图、用例图自动识别成本较高，第一版先以代码和用户输入为主

范围约束：

- 第一版优先支持 Java
- 第一版不强制实现 UML 图像识别
- 第一版重点保证代码度量和面向对象度量可用

应对策略：

- 先实现源码分析核心能力
- 将图形输入类需求转化为用户表单输入
- 对复杂指标给出计算说明，保证报告可解释
